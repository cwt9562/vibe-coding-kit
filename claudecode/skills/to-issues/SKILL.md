# 转换为 Issues

使用垂直切片（tracer bullets）将计划分解为可独立领取的问题。

## 流程

### 1. 收集上下文

从对话上下文中已有的内容开始工作。如果用户将问题引用（问题编号、URL或路径）作为参数传递，请从问题跟踪器中获取该问题并读取其完整正文和评论。

### 2. 探索代码库（可选）

如果尚未探索代码库，请先了解代码的当前状态。问题标题和描述应使用项目的领域术语词汇表，并尊重你所涉及区域的 ADR。

### 3. 起草垂直切片

将计划分解为 **tracer bullet** 问题。每个问题是一个薄而完整的垂直切片，贯穿所有集成层端到端，而非单一层的水平切片。

切片可以是 "HITL" 或 "AFK"。HITL 切片需要人工交互，例如架构决策或设计评审。AFK 切片可以在无需人工交互的情况下实现和合并。尽可能优先选择 AFK 而非 HITL。

<vertical-slice-rules>
- 每个切片提供一条狭窄但完整的路径，贯穿每一层（schema、API、UI、测试）
- 完成后的切片可以独立演示或验证
- 优先选择多个薄切片而非少数厚切片
</vertical-slice-rules>

### 4. 询问用户

将提议的分解以编号列表形式呈现。对于每个切片，展示：

- **标题**：简短描述性名称
- **类型**：HITL / AFK
- **阻塞于**：哪些其他切片（如果有）必须先完成
- **覆盖的用户故事**：这解决了哪些用户故事（如果源材料中有）

询问用户：

- 粒度是否合适？（太粗 / 太细）
- 依赖关系是否正确？
- 是否有任何切片需要合并或进一步拆分？
- 标记为 HITL 和 AFK 的切片是否正确？

迭代直到用户批准该分解方案。

### 5. 将问题发布到问题跟踪器

对于每个已批准的切片，向问题跟踪器发布一个新问题。使用下面的问题正文模板。这些问题被视为 AFK agent 可以处理的任务，因此除非另有指示，请使用正确的分类标签发布。

按依赖顺序发布问题（阻塞项优先），这样你可以在 "Blocked by" 字段中引用真实的问题标识符。

<issue-template>
## 父问题

问题跟踪器中父问题的引用（如果源是现有问题，否则省略此部分）。

## 要构建的内容

对该垂直切片的简明描述。描述端到端行为，而非逐层实现。

避免使用具体的文件路径或代码片段 — 它们很快就会过时。例外：如果原型产生的代码片段比文字描述更精确地表达了决策（状态机、reducer、schema、类型定义），请在此处内联包含，并简要说明它来自原型。只保留包含决策的部分 — 不需要可运行的演示，只保留关键内容。

## 验收标准

- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

## 阻塞于

- 阻塞工单的引用（如果有）

如果没有阻塞项，则填写 "无 - 可立即开始"。

</issue-template>

不要关闭或修改任何父问题。
