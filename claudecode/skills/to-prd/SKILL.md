本 skill 基于当前对话上下文和代码库理解生成 PRD。不要向用户提问 —— 直接综合已有信息即可。

issue 跟踪器和分类标签词汇应已提供给你 —— 如果尚未提供，请运行 `/setup-matt-pocock-skills`。

## 流程

1. 如果你尚未探索代码库，先了解其当前状态。在整个 PRD 中使用项目的领域术语词汇，并尊重你涉及区域内的任何 ADR（架构决策记录）。

2. 绘制完成该实现所需构建或修改的主要模块。主动寻找可提取出能独立测试的 deep module 的机会。

deep module（与 shallow module 相对）是指将大量功能封装在简单、可测试且极少变更的接口中的模块。

与用户确认这些模块是否符合预期。询问用户希望哪些模块编写测试。

3. 使用以下模板编写 PRD，然后发布到项目 issue 跟踪器。应用 `ready-for-agent` 分类标签 —— 无需额外分类。

<prd-template>

## 问题陈述

用户当前面临的问题，从用户视角出发。

## 解决方案

问题的解决方案，从用户视角出发。

## 用户故事

一份较长的、编号的用户故事列表。每个用户故事应采用以下格式：

1. 作为<角色>，我想要<功能>，以便<收益>

<user-story-example>
1. 作为手机银行客户，我想要查看我的账户余额，以便我能更明智地做出消费决策
</user-story-example>

用户故事列表应尽可能详尽，覆盖该功能的各个方面。

## 实现决策

已做出的实现决策列表。可包括：

- 将要构建/修改的模块
- 这些模块将要修改的接口
- 来自开发者的技术说明
- 架构决策
- 数据模型变更
- API 契约
- 具体交互方式

不要包含具体的文件路径或代码片段。它们可能会很快过时。

例外情况：如果原型产生了比文字描述更精确的表达决策的代码片段（状态机、reducer、schema、类型结构），可将其内联到相关决策中，并简要说明其来自原型。仅保留富含决策信息的部分 —— 不是可运行的演示，只保留关键内容。

## 测试决策

已做出的测试决策列表。包括：

- 良好测试的标准描述（只测试外部行为，不测试实现细节）
- 哪些模块将被测试
- 测试的已有参考（即代码库中类似的测试类型）

## 范围外

本 PRD 中不属于范围的描述。

## 补充说明

关于该功能的任何补充说明。

</prd-template>
