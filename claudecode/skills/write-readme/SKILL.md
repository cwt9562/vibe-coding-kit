# 中文 README 生成器

## 触发条件
当用户要求写 README、重写项目介绍、补全文档首页、优化开源仓库展示页时激活此技能。

## 核心原则
1. 先读代码和目录，再写 README；不要只凭项目名猜功能。
2. 面向中文开发者表达，技术名词保留英文，解释用自然中文。
3. 首页要解决三个问题：这是什么、为什么值得用、怎么快速上手。
4. 优先给出真实命令、真实路径、真实示例，少写空泛宣传语。
5. 如果项目已有 README，先保留有效信息，再补全缺失结构。
6. 如果项目包含官网或文档站，README 中必须给出统一、可点击的正式入口链接，避免 README / 官网 / Issue 模板地址不一致。
7. 开源项目可补充社交证明信号（Star、精选数量、更新时间、代表性场景），帮助访客快速判断项目活跃度和质量；内部项目则无需此类信息。
8. README 首屏应让读者在 30 秒内判断是否值得继续阅读：首句说清价值主张，接着说明目标用户或典型场景，最后给出最短安装/使用方式。不同项目类型的首屏呈现各有侧重，参见下方的分场景首屏模板。
9. 如果项目有官网、Issue 模板、镜像地址或社区入口，应将同类入口收敛为一处，减少用户在多个链接间跳转。内部项目同理，统一入口（如 Wiki / 文档站 / 内网地址）。
10. 如存在 badge、Hero 文案、官网首页等配套渠道，README 文案要与这些渠道保持口径一致，避免承诺和数据冲突。
11. **先判断项目类型，再选择合适的 README 模板和章节**：开源项目侧重社交证明、多场景示例及版本兼容性；内部项目侧重团队信息、部署运维和快速上手。
12. **内部项目要补全"谁能帮你"信息**：包括负责人/维护团队、on-call 方式、沟通渠道（如钉钉群、飞书群、企业微信）、故障上报入口。

## 分场景首屏模板

根据项目类型选择对应的首屏结构，不必照搬所有要素。

### 开源项目首屏
- **一句话价值主张**：明确解决什么问题，避免"强大/优雅/现代化"之类空词。
- **适合谁用**：点名目标用户或典型场景，例如"面向中文开发者""适合团队协作""适合零配置上手"。
- **最短安装方式**：给出一段可直接复制的命令，优先放 GitHub 原生地址；如有镜像，再作为补充。
- **统一入口**：官网、文档、Issue 征集、下载方式尽量收敛到 1 个正式入口，减少用户决策成本。
- **社交证明**：在不堆砌数据的前提下，展示 Star、精选数量、更新时间或代表性案例。

### 内部项目首屏
- **一句话说明**：这个项目/服务是做什么的，属于哪个团队或业务域。
- **快速启动**：最短路径让读者在本地跑起来，包括内网 registry、VPN 等前置条件。
- **负责人与联系方式**：维护者姓名、团队名称、沟通群（钉钉/飞书/企业微信）、on-call 方式。
- **统一入口**：内网文档站 / Wiki 地址、部署平台链接、监控面板链接。

### 工具/脚本首屏
- **一句话说明**：这个工具解决什么场景的问题。
- **适用场景**：什么情况下该用它，什么情况下不建议用。
- **一行安装**：pip install / npm install / curl 脚本等。
- **最短用例**：一条命令或一段代码展示核心用法。

## 工作流程
1. 检查项目结构、包管理文件、入口文件、测试目录和已有文档。
2. **判断项目类型**。根据以下信号判断：
   - 有 `LICENSE` 文件 + GitHub 远程仓库（`git remote -v` 指向 github.com）→ 开源项目
   - 其余情况（含内部 registry 配置、单文件/少文件工具、SDK/库、或无法明确判断）→ 内部项目
3. 提炼项目定位：目标用户、核心能力、和替代方案相比的差异点。
4. 根据项目类型，选择对应章节组合（必选 + 推荐 + 可选），优先补全"安装/环境准备 / 快速开始 / 使用示例 / 项目结构"。
5. 如果缺少真实命令，基于仓库中的 `package.json`、`pyproject.toml`、`Makefile`、`Cargo.toml` 等提取。
6. 用中文重写标题、摘要和亮点，避免"强大、优雅、现代化"这类空词。
7. 最后检查 README 是否能支持首次访问者在 30 秒内理解并开始使用。

## 输出要求
- 默认使用 Markdown。
- 标题层级清晰，emoji 适量，不要堆砌。
- 示例命令必须可复制。
- 如果信息不足，显式标注"待补充"，不要编造不存在的特性。

## 自检清单

### 通用（所有项目类型）
- 是否说明了项目解决的问题？
- 是否给出最短可运行示例？
- 是否覆盖安装和使用两个基本环节？
- 是否避免了 AI 腔和空泛营销语？
- 是否让目标读者一眼看懂？

### 开源项目
- 是否包含许可证信息和贡献方式？
- 是否提供了社交证明（Star、更新时间等）？
- 是否有统一的官方入口链接（官网/文档/Issue）？
- 是否提供了 API 文档链接或示例？
- 是否说明了版本兼容性要求？
- 如果有重大变更，是否有迁移指南？

### 内部项目
- 是否标注了负责人、团队和联系方式？
- 是否说明了内网环境准备（registry、VPN、依赖服务）？
- 是否提供了部署流程或监控入口？
- 是否有故障上报渠道？
- 工具/脚本场景：是否一句话说清了适用场景？安装方式是否足够短（一行搞定）？用例是否从"最短路径"开始？
- SDK/服务场景：是否提供了 API 文档链接或示例？是否说明了版本兼容性要求？
